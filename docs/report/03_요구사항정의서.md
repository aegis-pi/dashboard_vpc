### 2-2. 범위

| 범위 항목 | 내용 |
|-----------|------|
| **인프라 요구조건** | 클라우드 허브와 복수 엣지 공장 Spoke를 연결하는 멀티 공장 인프라를 구축해야 한다. 제어/관리 영역(2번 Control/Management VPC)과 사용자 조회·데이터 처리 영역(1번 Data/Dashboard VPC)은 네트워크 및 권한 경계로 분리되어야 한다. 1번 VPC는 Public, Private App, Private Data 계층을 갖추고 ALB, ECS Fargate Backend, Redis, RDS PostgreSQL을 포함해야 한다. |
| **글로벌/관리형 서비스 요구조건** | Dashboard는 VPC 내부 자원만으로 구성하지 않고 CloudFront, WAF, Route53, ACM, Cognito, S3 정적 호스팅, IoT Core, DynamoDB, EventBridge, Bedrock, Secrets Manager, CloudWatch/X-Ray 등 VPC 밖 글로벌·관리형 서비스를 목적별로 사용해야 한다. |
| **Dashboard 요구조건** | 본사 관제 담당자용 Risk Twin Dashboard를 제공해야 한다. Fleet Overview, Factory Detail, Environment, Infrastructure, Timeline, 일간 보고서 탭을 포함하고 공장별 현재 위험도와 원인을 우선 표시해야 한다. |
| **사용자/권한 요구조건** | 사용자는 Cognito 기반 인증 후 RDS PostgreSQL의 사용자·공장·권한 관계에 따라 자신에게 허용된 공장 데이터만 조회할 수 있어야 한다. 전체 공장 조회, 단일 공장 조회, 보고서 조회, 사용자/알림룰 관리 범위는 역할별로 분리되어야 한다. |
| **데이터 요구조건** | Edge Agent가 IoT Core로 전송한 `factory_state`와 `infra_state`를 S3 raw에 원본 보존하고, Lambda data processor가 DynamoDB LATEST/HISTORY와 S3 processed를 갱신해야 한다. Dashboard는 현재 상태는 DynamoDB LATEST, 최근 그래프는 DynamoDB HISTORY, 상세/감사/리포트는 S3 processed·reports prefix를 우선 조회해야 한다. |
| **실시간 요구조건** | DynamoDB Streams, Lambda notifier, Redis Pub/Sub, ECS WebSocket을 통해 공장 상태 변화가 Dashboard Web에 실시간 push되어야 한다. WebSocket 장애 시 30초 polling fallback을 제공해야 한다. |
| **보안 요구조건** | Dashboard는 Spoke K3s API, EKS admin API, ArgoCD admin API, Tailscale 관리망, S3 raw 전체 권한에 직접 접근할 수 없어야 한다. 모든 사용자 진입점은 HTTPS·인증을 강제하고, Secret/Token/Private Key는 코드·문서에 기록하지 않아야 한다. |
| **CI/CD 요구조건** | 인프라·클러스터·빌드·배포 각 영역의 책임 경계를 명확히 분리하고, 수동 개입 없이 자동화된 파이프라인을 통해 변경 가능해야 한다. 클러스터 앱 상태 변경은 GitOps 경로를 사용해야 한다. |
| **운영/비용 요구조건** | 데모 운영 패턴에서는 `build-data-dashboard.sh`로 생성하고 `destroy-data-dashboard.sh`로 종료해 비용을 제어해야 한다. 새 AWS 상시 리소스 추가 시 비용 기준 문서가 함께 갱신되어야 한다. |
| **사업 외 범위(Out of Scope)** | 공장 신축·OT 네트워크 물리 인프라, 사내 BCP/DR 수립, 발주처 SSO IdP 본격 federation은 본 사업 범위 외로 한다. |

## 3. 사업 요구사항

### 3-1. 시스템 구축 요구사항

| 요구사항 ID | 분류 | 항목 | 요구사항 내용 | 중요도 |
|-------------|------|------|--------------|:------:|
| SFR-001 | Infrastructure | 전체 구조 | `factory-a/b/c` Spoke를 중앙 Hub와 Data/Dashboard VPC로 통합 관리할 수 있어야 한다. Control/Management VPC는 배포·제어·운영 관측, Data/Dashboard VPC는 데이터 처리 결과 조회와 사용자 Dashboard 제공을 담당해야 한다. | H |
| SFR-002 | Infrastructure | 1번 VPC | 1번 VPC는 Public, Private App, Private Data Subnet으로 구성한다. Public에 HTTPS ALB·NAT GW 1개, Private App에 ECS Fargate Backend·ElastiCache Redis·Lambda notifier 접근 경계, Private Data에 RDS PostgreSQL을 배치한다. | H |
| SFR-003 | Infrastructure | VPC 밖 글로벌/관리형 서비스 | Dashboard Web은 S3 정적 SPA + CloudFront로 제공한다. S3는 Public 차단, CloudFront OAC만 허용. WAF는 외부 진입점 보호, Route53은 신규 도메인 위임과 `dashboard.<domain>`·`api.<domain>` alias 관리, ACM은 CloudFront용 us-east-1과 ALB용 regional 인증서 분리. | H |
| SFR-004 | Infrastructure | 데이터 처리 경로 | Edge Agent → IoT Core → IoT Rule(S3 raw 적재) → Lambda data processor(정규화·Risk·pipeline·LATEST/HISTORY·processed 생성) → DynamoDB Streams → Lambda notifier → Redis Pub/Sub → WebSocket 경로를 구현한다. | H |
| SFR-005 | Infrastructure | Server | 운영형 Spoke는 다중 노드 K3s로 단일 노드 장애 시 대체 노드에서 서비스가 계속되어야 한다. Dashboard Backend는 FastAPI 기반 ECS Fargate 컨테이너로 운영되어 REST/WebSocket/RDS/DynamoDB/S3/Redis/Bedrock 연동을 제공한다. | H |
| SFR-006 | Infrastructure | Network | 허브-Spoke 제어 통신은 암호화된 전용 채널을 사용하며 사용자 Dashboard 접근망과 분리한다. Dashboard는 Tailscale/VPN에 직접 의존하지 않고 CloudFront/ALB HTTPS와 Cognito 인증으로만 접근을 허용한다. VPC peering/TGW로 조회망과 제어망을 직접 연결하지 않는다. | H |
| SFR-007 | Infrastructure | Container | GitOps 선언적 방식으로 공장별 컨테이너 워크로드를 중앙 배포·동기화·롤백할 수 있어야 한다. Dashboard Backend 이미지는 ECR에 저장하고 ECS Service가 ALB target group 뒤에서 안정 실행되어야 한다. | H |
| SFR-008 | Infrastructure | Common | 빌드·배포 파이프라인은 수동 개입 없이 자동화되어야 한다. 공장별 설정은 버전 관리 시스템에서 독립 관리되며 공장 추가 시 기존 구조 변경 없이 확장 가능해야 한다. | H |

### 3-2. Dashboard 및 사용자별 조회 범위 요구사항

| 사용자/역할 | 조회 가능 범위 | 수행 가능 기능 | 제한 사항 |
|-------------|---------------|----------------|-----------|
| **본사 관제 관리자(Admin)** | 전체 공장 Fleet, 공장별 상세, Risk/환경/인프라 이력, Timeline, 일간 보고서, 사용자·공장·권한 메타데이터 | 전체 공장 위험 현황 확인, 사용자/권한 매핑 관리, 알림룰 설정, 감사 로그 확인, 임시 사용자 비활성화 | Dashboard에서 ArgoCD/EKS/Spoke API를 직접 제어하지 않음. 배포·클러스터 변경은 GitOps/운영 경로로만 수행 |
| **본사 관제 담당자(Operator)** | RDS `user_factory_access`에 매핑된 공장 목록과 해당 공장의 상세 화면·보고서·최근 이벤트 | 위험 공장 식별, 원인 확인, 보고서 열람, 운영 대응 판단 | 다른 공장 데이터와 사용자/권한 관리 화면은 조회 불가 |
| **현장 운영자(Local Operator)** | 자신이 담당하는 단일 공장 또는 지정 공장 그룹의 현재 상태, 장치/노드/워크로드 상태, Timeline | 현장 이상 여부 확인, 센서/장치/워크로드 이상 파악 | 전체 Fleet 통계, 타 공장 상세, 전역 사용자 관리, 클라우드 인프라 설정 조회 불가 |
| **배포 담당자/시스템 관리자** | 운영 관측 도구(Grafana, ArgoCD, CloudWatch 등)와 필요 시 지정된 Dashboard 공장 상태 | 배포 상태 확인, 장애 원인 분석, 운영 로그·메트릭 확인 | 사용자용 Dashboard가 운영 제어 plane이 되어서는 안 됨. Dashboard 권한은 별도 승인된 범위로 제한 |
| **발표/검토용 임시 사용자(Viewer)** | 지정된 공장 또는 데모용 전체 Fleet의 read-only 화면 | 화면 확인, 보고서 열람 | 쓰기, 권한 변경, 알림룰 변경, 운영 API 접근 불가. 시연 후 계정 disable 필요 |

### 3-4. 데이터 및 저장소 요구사항

| 요구사항 ID | 항목 | 요구사항 내용 | 중요도 |
|-------------|------|--------------|:------:|
| DAR-001 | S3 raw | IoT Core 수신 원본 JSON은 `raw/{factory_id}/{source_type}/yyyy=...` 경로에 변경 불가 형태로 장기 보존. 감사·재처리·리포트 근거로 사용 가능해야 하며 Dashboard 기본 조회 경로로 사용하지 않음. 최소 180일 보존. | H |
| DAR-002 | S3 processed | Lambda data processor의 Risk·환경·인프라 처리 결과는 `processed/` prefix에 장기 보존. Dashboard는 상세 이력·감사·리포트 입력에 processed 데이터 사용 가능. | H |
