## 1. 기능 요구사항 (Functional Requirements)

시스템이 수행해야 하는 구체적인 동작·기능·프로세스에 대한 요구사항.

### 1.1 인프라 및 네트워크 (Infrastructure & Network)

| ID | 요구사항 명 | Risk Twin 기반 실현 내용 | 구현/검증 도구 |
|----|-------------|--------------------------|----------------|
| FR-INF-01 | IaC 기반 인프라 구축 | Data/Dashboard VPC, ALB, ECS, RDS, Redis, IAM 등 1번 VPC 전 구성요소를 Terraform 코드로 정의해 인적 오류를 방지하고 형상을 상시 관리 | Terraform |
| FR-INF-02 | 워크스트림 망 분리 | 2번 Control/Management VPC와 1번 Data/Dashboard VPC를 네트워크·권한 경계로 분리하고 VPC peering/TGW로 직접 연결하지 않음 | AWS VPC, IAM |
| FR-INF-03 | 1번 VPC 3-Tier 구성 | Public(ALB·NAT)·Private App(ECS·Redis)·Private Data(RDS) 서브넷으로 계층화하여 사용자 진입점과 데이터 저장 계층을 분리 | AWS VPC, Subnet, SG |
| FR-INF-04 | 글로벌/관리형 서비스 사용 | Dashboard Web은 S3 + CloudFront(OAC)로 제공하고, WAF·Route53·ACM·Cognito·EventBridge·Bedrock 등 VPC 밖 관리형 서비스를 목적별로 사용 | CloudFront, WAF, Route53, ACM, Cognito |
| FR-INF-05 | HTTPS 전용 진입점 | 사용자 진입점(CloudFront, ALB)은 HTTPS만 허용하고 HTTP는 자동 redirect | ACM, CloudFront, ALB Listener |
| FR-INF-06 | 도메인 분리 | Admin UI는 `*.minsoo-tech.cloud` 유지, Dashboard는 신규 도메인(`dashboard.<domain>`·`api.<domain>`)으로 분리하여 운영/사용자 진입점 혼동 차단 | Route53, ACM (us-east-1, regional) |
| FR-INF-07 | 비용 통제 운영 절차 | 데모 운영 시 `build-data-dashboard.sh`로 생성·`destroy-data-dashboard.sh`로 종료해 NAT/ALB/RDS/Redis 상시 비용 통제 | Shell Script, Terraform |

### 1.2 데이터 파이프라인 (Data Pipeline)

| ID | 요구사항 명 | Risk Twin 기반 실현 내용 | 구현/검증 도구 |
|----|-------------|--------------------------|----------------|
| FR-DATA-01 | Edge → IoT Core 발행 | Edge Agent가 `factory_state`·`infra_state` 토픽을 X.509 mTLS로 IoT Core에 발행하여 공장 상태를 안전하게 전송 | AWS IoT Core, X.509 |
| FR-DATA-02 | 원본 데이터 불변 보존 | IoT Rule이 수신 JSON을 `raw/{factory_id}/{source_type}/yyyy=...` 경로로 S3 raw에 적재. 감사·재처리·리포트 입력 근거로 사용 | IoT Rule, S3 |
| FR-DATA-03 | Risk 정규화/계산 | Lambda data processor가 원본을 정규화하고 센서·AI·stale·pipeline 상태를 종합해 0~100 Risk Score를 산출. **점수가 높을수록 안전**이며 구간은 `안전 100~85 / 경고 84~50 / 위험 49~0`으로 분류 | AWS Lambda (Python) |
| FR-DATA-04 | LATEST/HISTORY 저장 | 공장별 현재 상태는 DynamoDB LATEST(`pk=FACTORY#{id}`, `sk=LATEST`) 단일 item으로, 단기 시계열은 HISTORY item으로 저장(Risk/환경 30s, 인프라 20s, TTL 24h) | Amazon DynamoDB |
| FR-DATA-05 | 처리 결과 장기 보존 | Risk·환경·인프라 처리 결과를 S3 `processed/` prefix에 저장해 상세 이력·감사·리포트 입력에 사용 | S3 |
| FR-DATA-06 | 실시간 변화 푸시 | DynamoDB Streams → Lambda notifier → ElastiCache Redis Pub/Sub → ECS WebSocket 경로로 공장 상태 변화를 Dashboard에 push (1~2초 목표) | DDB Streams, Lambda, Redis, ECS |
| FR-DATA-07 | 일간 LLM 보고서 생성 | EventBridge Scheduler가 매일 09:00 KST에 report-generator Lambda를 실행해 지난 24시간 추세를 Bedrock Claude 3 Haiku로 한국어 Markdown 보고서로 생성·저장 | EventBridge, Lambda, Bedrock |
| FR-DATA-08 | 보고서 메타데이터 관리 | 생성 보고서는 `reports/<YYYY-MM-DD>/<factory_id>.md`에 저장하고 메타데이터(`report_date`·`factory_id`·`s3_key`·`summary`·`generated_at`·`model_id`)는 `aegis-daily-report` 테이블에 기록 | S3, DynamoDB |
| FR-DATA-09 | 데이터 수명주기 계층 | Hot(LATEST/HISTORY) / Warm(processed) / Cold(raw, reports) 계층 구분과 IA/Glacier 전환 정책을 적용해 보존성과 비용을 균형 | S3 Lifecycle, DDB TTL |

### 1.3 Dashboard 워크로드 (Dashboard Workload)

| ID | 요구사항 명 | Risk Twin 기반 실현 내용 | 구현/검증 도구 |
|----|-------------|--------------------------|----------------|
| FR-DASH-01 | Fleet Overview 화면 | 진입 시 전체 공장 수, 위험/주의/안전 수, 데이터 지연 수, 공장별 Risk 카드를 위험도→점수→pipeline abnormal→최신성 순으로 표시 | React, FastAPI, DDB LATEST |
| FR-DASH-02 | Factory Overview 화면 | 공장 상세 첫 화면에 현재 Risk Score/Level, Risk Twin 도메인 2개(Environmental / Infrastructure)의 sub-score, pipeline status, 노드 요약, 주요 원인, 환경 센서·AI 결과, 워크로드·장치 요약을 단일 화면에 제공 | React, FastAPI, DDB LATEST |
| FR-DASH-03 | Environment 추세 화면 | Risk Score, 온도(temperature_celsius_avg), 습도(humidity_percent_avg), 기압(pressure_hpa_avg), 화재/넘어짐/굽힘 AI score(fire/fall/bend_score)의 1h/2h/24h 추세를 30초 bucket으로 표시. abnormal_sound 텍스트 이벤트는 Timeline에 함께 표기 | React Chart, DDB HISTORY |
| FR-DASH-04 | Infrastructure 화면 | 노드 Ready/CPU/Memory/Disk usage 및 network_reachability, workload status·restart_count, 장치 가용성(BME280·Camera·Microphone), heartbeat publish_sequence, pipeline 상태를 20초 bucket으로 표시 | React Chart, DDB HISTORY |
| FR-DASH-05 | Timeline 화면 | Risk level 변화, 주요 원인, pipeline status, node ready, workload restart, device(BME280/Camera/Mic) 가용성 변화, abnormal_sound 텍스트 이벤트를 시간순으로 통합 표시해 원인 추적성 확보 | React, DDB HISTORY, S3 processed |
| FR-DASH-06 | 일간 보고서 열람 | 날짜·공장 기준으로 보고서를 조회·Markdown 렌더링. 권한 없는 공장 보고서는 차단 | React Markdown, S3, DDB |
| FR-DASH-07 | WebSocket 실시간 갱신 | DDB Streams 이후 push 1~2초 목표로 Dashboard를 갱신하고, 장애 시 30초 polling fallback 제공 | ECS Fargate WebSocket, Client SDK |
| FR-DASH-08 | 사용자별 조회 범위 | Cognito JWT 검증 후 RDS `user_factory_access`에 매핑된 공장만 응답에 포함. 미인가 공장 직접 호출은 403 차단 | FastAPI, Cognito, RDS, Redis cache |
| FR-DASH-09 | 역할 분리 | 사용자 역할(관리자 / 일반 조회자)에 따라 접근 가능한 화면과 API 범위를 분리하여 운영·열람 책임을 분리 | FastAPI RBAC, Cognito Group |
| FR-DASH-10 | Alerts 화면 | Risk Score가 경고(84~50)·위험(49~0) 구간으로 진입하거나 pipeline·node·workload·device 이상이 감지되면 Dashboard의 Alerts 화면에 항목을 표시. 외부 채널(Slack/Email) 송신은 하지 않음. severity(crit/warn/info)·domain(environmental/infrastructure/operational)·site로 필터링하고 `open → ack → resolved` 흐름과 acknowledge/snooze/assign 액션을 제공 | React Alerts Page, FastAPI, DDB Streams |
| FR-DASH-11 | 감사 로그 | 로그인·권한 변경·알림룰 변경·보고서 열람을 감사 로그로 기록·조회 | FastAPI, RDS PostgreSQL |
| FR-DASH-12 | 메타데이터 관리 | Admin이 사용자·공장·`user_factory_access`·알림룰을 CRUD하여 조직 변화에 대응 | FastAPI, RDS PostgreSQL |

## 2. 비기능 요구사항 (Non-Functional Requirements)

시스템의 품질 속성(보안성·성능·제약사항 등)에 대한 요구사항.

### 2.1 보안 (Security)

| ID | 요구사항 명 | Risk Twin 기반 실현 내용 | 구현/검증 도구 |
|----|-------------|--------------------------|----------------|
| NFR-SEC-01 | 사용자 인증 | Cognito Hosted UI OIDC PKCE 흐름, Self sign-up 비활성화, 관리자 생성 사용자만 허용, MFA(TOTP) 필수 | AWS Cognito, TOTP |
| NFR-SEC-02 | 사용자 인가 | Cognito JWT를 앱 레벨 검증하고 RDS 권한 모델로 API 응답을 필터링. 권한 캐시는 Redis TTL로 일관성 유지 | FastAPI, RDS, ElastiCache Redis |
| NFR-SEC-03 | 최소 권한 IAM | Dashboard API/Lambda는 LATEST/HISTORY·processed·reports·메타데이터에 한정된 최소 IAM Task Role/Lambda Role만 보유 | AWS IAM Task Role |
| NFR-SEC-04 | 권한 격리(통제 plane 차단) | Dashboard Web/API는 Spoke K3s API, EKS admin API, ArgoCD admin API, Tailscale 관리망에 직접 접근 불가 | IAM, SG, VPC 분리 |
| NFR-SEC-05 | 웹 취약점 방어 | CloudFront에 WAF를 배치해 SQL Injection, XSS, Bot Control 패턴을 상시 검증·차단 | AWS WAF (Managed Rules) |
| NFR-SEC-06 | 데이터 암호화 | 저장 암호화(S3 SSE-S3/KMS, RDS·DDB 기본 암호화)와 전송 암호화(TLS 1.2 이상)로 무결성·기밀성 보장 | AWS KMS, ACM, TLS |
| NFR-SEC-07 | 비밀정보 관리 | DB 비밀번호·Redis AUTH token·Cognito client secret·private key·MFA OTP는 코드·문서·LLM 프롬프트에 기록 금지. Runtime은 Secrets Manager 참조 | AWS Secrets Manager |
| NFR-SEC-08 | 감사 추적 | 공장 원본 데이터를 변경 불가능한 S3 raw에 보존하고 사용자 활동 감사 로그는 RDS에 90일 이상 보존해 추적성 확보 | S3 Versioning, RDS Audit |
| NFR-SEC-09 | 네트워크 보안 | RDS는 ECS task SG에서만 5432 허용, Redis는 ECS task·Lambda notifier SG에서만 접근하며 transit encryption + AUTH token 적용 | AWS Security Group |
| NFR-SEC-10 | 보안 점검 | 사업 종료 전 정적(Trivy/Checkov)·동적(ZAP 또는 동등)·IAM 권한 점검 결과서를 제출하고 발견 사항 모두 조치 또는 합의 | Trivy, Checkov, ZAP |
| NFR-SEC-11 | LLM 프롬프트 안전 | Bedrock 프롬프트에 Secret/token/private key/MFA OTP/계정 세부정보 포함 금지. Bedrock IAM은 InvokeModel과 최소 데이터 읽기 권한만 부여 | Lambda 정책, IAM 분리 |
| NFR-SEC-12 | 발주처 보안정책 준수 | 발주처 정보보안실의 정보보호 지침(ID/PW 정책, 세션 타임아웃, 접근 IP 화이트리스트 등)을 분석 단계에서 수렴해 설계 반영 | 분석 산출물 |

| NFR-PER-04 | WebSocket 동시 접속 | 단일 ECS Service 기준 100 conn/task 안정 | 부하 시험 도구 |
| NFR-PER-05 | Frontend 초기 로딩 | CloudFront edge → 첫 화면 interactive < 3s (Wi-Fi 50Mbps) | Lighthouse |
| NFR-PER-06 | 일간 보고서 생성 시간 | factory 1개 기준 report-generator Lambda 실행 시간 < 90s | Lambda Duration Metric |
| NFR-PER-07 | 확장성 | 공장 N개 증가 시 IaC 변수와 GitOps 매니페스트만으로 기존 구조 변경 없이 N=10까지 신규 공장 추가 가능 | Terraform, ArgoCD ApplicationSet |

### 2.3 제약 사항 (Constraints)

| ID | 요구사항 명 | Risk Twin 기반 실현 내용 | 구현/검증 도구 |
|----|-------------|--------------------------|----------------|
| NFR-CON-01 | 클라우드 제약 | AWS 단일 클라우드만 사용하며 멀티 클라우드는 본 사업 범위 외 | 사업 합의 |
| NFR-CON-02 | 지역(Region) 제한 | 기본 운영 리전은 `ap-south-1`. CloudFront/ACM은 us-east-1 예외. 허용되지 않은 리전 접근은 IAM 정책으로 차단 | AWS IAM 정책 |
| NFR-CON-03 | 명명 규칙 | `AEGIS-[resource]-[feature]-[zone]` 명명 규칙을 모든 AWS 리소스에 적용 | Terraform 변수, Tag 정책 |
| NFR-CON-04 | 워크스트림 분리 | 본 사업 범위는 1번 Data/Dashboard VPC(워크스트림 B). 워크스트림 A(Control/Management) 자산은 명시적 지시 없이는 수정·삭제 금지. 합류 지점(IoT Core Rule, S3, ECR, DynamoDB, GitHub) 변경은 ADR로 남김 | ADR 문서 |
| NFR-CON-05 | LLM 모델 정책 | 일간 보고서는 Bedrock Claude 3 Haiku 우선. 대체 모델 사용 시 ADR로 근거 남김 | Bedrock, ADR |
| NFR-CON-06 | 데이터 주권 | 별도 지정이 없으면 모든 데이터는 발주처 AWS 계정 내에 보관하며 제3자 SaaS로 반출 금지 | AWS Account 경계 |
| NFR-CON-07 | 내부자 통제 | 운영자의 API 직접 호출 권한을 제한하고 사전 정의된 RBAC·IAM Role로만 접근 강제 | K8s RBAC, AWS IAM |
| NFR-CON-08 | 개인정보 최소화 | 사용자 식별자는 Cognito `sub`만 RDS에 저장. 이름·이메일은 표시 목적 외 저장 최소화 | RDS 스키마 |
| NFR-CON-09 | 라이선스 | 사용된 OSS 라이선스 목록과 호환성 점검 결과서 제출 | OSS Audit |
| NFR-CON-10 | 사업 외 범위 | 공장 신축·OT 네트워크 물리 인프라, 사내 BCP/DR 수립, IdP federation 본격 도입은 본 사업 범위 외 | 사업 합의 |
| NFR-CON-11 | Risk Twin 도메인 범위 | Risk Twin은 actual IoT 데이터(`factory_state` 센서·AI score, `infra_state` 노드·workload·device)로 산출 가능한 **Environmental + Infrastructure 2개 도메인**으로 한정. 제조 OEE 지표(throughput, OEE, defect rate, vibration), Operational 도메인, PM2.5/air exchange, Power draw, PLC/Servo, Production lines, OEE/MTTR 집계는 actual data에 매핑 대상이 없어 본 사업 범위 외 | IoT 데이터 스키마 |
