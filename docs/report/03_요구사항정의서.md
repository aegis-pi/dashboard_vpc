## 1. 문서 개요

### 1.1 작성 목적

- 본 문서는 **발주처가 MSP 수행사에 위탁하는 Aegis-Pi Risk Twin 구축 사업**의 요구사항을 식별·정의하고, 이를 사업 수행 전 기간에 걸쳐 변경 통제 가능한 형태로 관리하기 위한 기준 문서이다.
- 본 문서의 요구사항은 사업 착수 시점의 **계약 범위(Scope of Work)** 와 **인수 기준(Acceptance Criteria)** 의 근거가 되며, 추가/변경 요청은 별도의 변경관리 절차(CR; Change Request)를 통해서만 반영된다.

### 1.2 적용 범위

| 구분 | 포함 여부 | 비고 |
|------|---------|------|
| 본사 통합 관제용 멀티 공장 Risk Twin Dashboard 구축 | 포함 | 1차 사용자: 본사 관제 담당자 |
| AWS Cloud Hub(Control / Management VPC) 및 Spoke K3s 운영 환경 | 포함 | 운영형 1개(`factory-a`) + 테스트베드 2개(`factory-b/c`) |
| Data / Dashboard VPC 및 사용자별 권한 분리 체계 | 포함 | 본 RFP의 핵심 신규 구축 범위 |
| 일간 LLM 보고서 자동 생성·열람 기능 | 포함 | Bedrock Claude 3 Haiku 사용 |
| 공장 신축 및 OT 네트워크 물리 인프라 | 제외 | 발주처 시설팀이 별도 수행 |
| 현장 노동·안전 매뉴얼, 사내 BCP/DR 수립 | 제외 | 시스템 산출물에 연계 권고만 제공 |

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| Risk Twin | 공장 환경·인프라 상태를 실시간 수집·정규화하여 0–100 Risk Score와 `안전/주의/위험` 상태로 환산해 표시하는 디지털 트윈 |
| Hub | AWS 상의 중앙 EKS 클러스터 및 관리형 서비스 집합 (Control / Management VPC) |
| Spoke | 공장 단위의 K3s 클러스터 (현장 Raspberry Pi 또는 VM) |
| Safe-Edge | 단일 공장 내 다중 노드 K3s 기반 생존형 엣지 구성 |
| Pipeline Status | Edge → IoT Core → S3 → Lambda → DynamoDB 흐름의 정상/지연/중단 판정 결과 |
| Stale | 정의된 임계 시간 동안 신규 데이터 도달이 없어 신뢰할 수 없다고 판단된 상태 |
| LATEST / HISTORY | DynamoDB 단일 테이블 내 최신 상태(item)와 시계열(item) 패턴 |

## 2. 사업 개요

### 2.1 사업 목적

- Safe-Edge 기반의 단일 공장 생존형 엣지 운영 체계를 본사 단위 **멀티 공장 중앙 관제 구조**로 확장한다.
- 본사 관제 담당자가 여러 공장의 위험 상태·원인·데이터 신뢰도·최근 변화·일간 보고서를 **단일 화면 흐름** 안에서 식별하고 대응할 수 있게 한다.
- 운영 통제 영역(클러스터·배포·VPN)과 사용자 조회 영역(Dashboard)을 **네트워크·권한 경계로 분리**하여 사용자용 화면이 운영 control plane을 겸하지 않도록 한다.

### 2.2 사업 기간 및 단계

| 단계 | 기간 | 주요 산출물 |
|------|------|-------------|
| 착수 | 계약일 + 0주 | 착수보고서, 본 요구사항정의서 v1.x, WBS |
| 분석·설계 | 계약일 + 1~3주 | AS-IS/TO-BE, 상세 아키텍처, 화면 설계서, 데이터 모델, 보안 설계 |
| 구축·이행 | 계약일 + 3~7주 | IaC, 컨테이너 이미지, GitOps 매니페스트, Dashboard Web/API |
| 통합 시험·검수 | 계약일 + 7~9주 | 통합시험결과서, 보안 점검 결과서, 성능 측정 결과서 |
| 인수 | 계약일 + 9주 | 인수확인서, 운영 인계서, 비용 기준서 |

> 총 사업 기간: 계약일로부터 60일 (영업일 기준, 공휴일 제외)

| 인수 인계 | 운영 절차서, 장애 대응 가이드, 권한 관리 가이드, 인수교육 (2회) |

## 3. 현황 분석 및 목표 시스템

### 3.1 현황(AS-IS) 요약

- `factory-a` 단일 공장에 Raspberry Pi 3노드 K3s 클러스터로 Safe-Edge가 운영 중이며, 노드 장애 시 워크로드 failover가 검증되어 있다.
- AWS 상에 EKS 기반 Hub, IoT Core, Foundation(S3·ECR·IAM), Admin UI(ArgoCD/Grafana)가 활성화되어 있으나, **본사 관제 담당자가 직접 사용할 Risk Twin Dashboard는 미구축** 상태이다.
- 현재는 운영자가 ArgoCD/Grafana UI를 통해 클러스터 관점으로 상태를 확인하며, 공장 단위 위험도·원인·일간 추세는 사용자 친화적으로 제공되지 못한다.

### 3.2 목표(TO-BE) 요약

- 본사 관제 담당자용 Risk Twin Dashboard와 1번 Data / Dashboard VPC를 신규 구축한다.
- 사용자는 Cognito 인증 + RDS 권한 모델로 **자신에게 허용된 공장**만 조회한다.
- 데이터는 `Edge Agent → IoT Core → S3 raw → Lambda → DynamoDB(LATEST/HISTORY) + S3 processed → Dashboard` 경로로 흐르며, DynamoDB Streams 기반 WebSocket push로 준실시간 갱신된다.
- LLM 일간 보고서가 매일 09:00 KST에 자동 생성·열람된다.

### 3.3 시스템 구성(개요)

```
[사용자 PC/모바일]
  └ HTTPS ─▶ CloudFront (WAF) ─▶ S3 Static SPA
                              └▶ ALB ─▶ ECS Fargate (FastAPI / WebSocket)
                                        ├─▶ RDS PostgreSQL    (사용자/공장/권한)
                                        ├─▶ ElastiCache Redis (캐시/Pub-Sub)
                                        ├─▶ DynamoDB         (LATEST/HISTORY/Report)
                                        └─▶ S3 (processed / reports)
                                              ▲
[Edge Agent] ─MQTT─▶ IoT Core ─Rule─▶ S3 raw ─▶ Lambda processor ─┘
                                                        │
                                              EventBridge 09:00 KST
                                              ─▶ Lambda report-generator
                                              ─▶ Bedrock (Claude 3 Haiku)
```

## 6. 가정 및 제약 (Assumptions & Constraints)

- 발주처는 사업 착수 시점에 AWS 계정, 결제 한도, Route53 hosted zone, 신규 도메인 위임 권한을 MSP에 제공한다.
- 발주처 정보보안실은 분석 단계 종료 전까지 보안 지침서를 MSP에 전달한다.
- 본 사업 기간 동안 `factory-a` 운영형 Spoke는 시연 가능한 상태로 유지된다.
- 메시지율, DDB HISTORY 비용, 시계열 쿼리 지연, 동시 사용자 수, 컴플라이언스 요구가 사업 기간 중 사전 합의된 임계값을 넘는 경우, Phase 2~4(Timestream / Kinesis / OpenSearch / Multi-AZ / SageMaker / IdP federation / Security Hub / GuardDuty / PrivateLink / Athena·Glue)는 별도 사업으로 분리한다.
