| **기타 서비스 요구조건** | 공장별 위험 수준을 실시간으로 정량화하고, 본사 관제 담당자가 이를 한 화면에서 확인·대응할 수 있는 Risk Twin Dashboard와 일간 보고서를 제공해야 한다 |

## 3. 사업 요구사항

### 3-1. 시스템 구축 요구사항

| 분류 | 항목 | 요구사항 내용 |
|------|------|--------------|
| **Infrastructure** | Infrastructure | 시스템은 클라우드 기반 중앙 허브를 통해 복수의 엣지 공장을 단일 관제 체계로 통합 관리할 수 있어야 한다. 데이터 수집·처리·보존·조회를 담당하는 중앙 관제 환경은 운영망과 분리된 별도 영역으로 구성되어야 한다. 클라우드 인프라는 복수의 가용영역에 걸쳐 단일 장애점(SPOF) 없이 운영되어야 한다. 제어망과 데이터/관제망은 네트워크 레벨에서 물리적으로 분리되어야 한다. 공장 센서 데이터를 클라우드로 실시간 수집·저장·처리할 수 있는 데이터 파이프라인을 갖추어야 한다. 모든 사용자 및 관리자 진입점은 암호화된 보안 통신을 통해서만 접근 가능해야 한다 |
| | Server | 운영형 엣지 Spoke는 다중 노드로 구성되어 단일 노드 장애 시에도 서비스를 무중단으로 유지해야 한다. 테스트베드형 Spoke는 운영형과 동일한 워크로드를 VM 환경에서 검증할 수 있어야 한다. 각 공장 Spoke는 허브와의 네트워크 단절 상황에서도 로컬 기능을 유지해야 한다. 중앙 관제 환경의 데이터 처리·조회·전달 컴포넌트는 본사 관제 담당자가 위험 상태를 안정적으로 확인할 수 있도록 무중단을 지향하는 형태로 운영되어야 한다 |
| | Network | 허브와 각 공장 Spoke 간 제어 통신은 암호화된 전용 채널(VPN)을 통해야 하며 관제 접근망과 분리되어야 한다. 공장 현장 데이터는 인증된 안전한 메시지 채널을 통해 클라우드로 전송되어야 한다. 수집된 원본 데이터는 자동으로 변경 불가능한 보존 저장소에 적재되어야 한다. 본사 관제 화면 접근은 웹 방화벽(WAF)의 보호하에서만 허용되어야 하며, 운영 관리 진입점과 도메인·접근 경로가 분리되어야 한다 |
| | Container | 시스템은 선언적 GitOps 방식으로 모든 공장의 컨테이너 워크로드를 중앙에서 배포·동기화·롤백할 수 있어야 한다. 엣지 노드는 스토리지 장애에 대비한 데이터 복제 기능을 갖추어야 한다. 시스템은 노드 이상을 자동으로 감지하고 워크로드를 대체 노드로 이전(Failover)할 수 있어야 한다. 테스트베드형 공장은 모의 데이터를 발행하여 전체 파이프라인을 검증할 수 있어야 한다. 본사 관제 담당자용 Dashboard는 컨테이너 형태로 중앙 관제 환경의 클라우드에서 운영되어야 한다 |
| | Common | 소프트웨어 빌드·배포 파이프라인은 수동 개입 없이 자동화되어야 한다. 각 공장별 배포 설정과 중앙 관제 환경의 형상은 버전 관리 시스템에서 독립적으로 관리되어야 하며, 공장 추가 시 기존 구조를 변경하지 않고 확장 가능해야 한다 |

### 3-2. 보안 요구사항

| 항목 | 요구사항 내용 |
|------|--------------|
| **시스템 감사** | 시스템은 공장에서 수집된 모든 원본 데이터를 변경 불가능한 형태로 전량 보존하여 감사 추적이 가능해야 한다. 동일 데이터의 중복 수집·처리를 방지하는 멱등성을 보장해야 한다. 원본 데이터는 가공 결과(처리 데이터)가 삭제된 이후에도 독립적으로 보존되어 재처리 및 일간 보고서 근거로 활용 가능해야 한다. 본사 관제 담당자의 주요 조회·열람 행위는 사후 분석이 가능한 형태로 기록되어야 한다 |
| **시스템 보안** | 본사 관제 Dashboard는 운영 클러스터의 관리 API에 직접 접근할 수 없어야 하며, 사용자별 조회 범위로 제한된 데이터 조회 전용 인터페이스만을 통해 운영되어야 한다. 제어망과 관제망은 네트워크 및 권한 경계로 완전히 분리되어야 한다. 자동화 파이프라인(CI)은 운영 클러스터에 직접 명령을 실행할 수 없어야 하며, 반드시 GitOps 채널을 통해서만 클러스터 상태 변경이 이루어져야 한다. 인증 정보(Secret, Access Key, Token 등)는 코드 저장소 및 문서에 포함되어서는 안 된다 |
| **보안솔루션** | 공장 제어망과 본사 관제 접근망은 별도의 보안 채널로 분리 운영되어야 한다. 본사 관제 Dashboard 접근은 WAF와 다중 인증(MFA)을 포함한 사용자 인증 체계로 보호되어야 한다. 각 공장 IoT 장치는 개별 인증서 기반으로 인증되어야 하며 타 공장 데이터 채널에 접근할 수 없어야 한다. 인증된 본사 관제 사용자라 하더라도 자신에게 허용된 공장 외의 공장 데이터에는 접근할 수 없어야 한다 |

### 3-3. 모니터링

| 항목 | 요구사항 내용 |
|------|--------------|
| **Metric 정의** | 시스템은 각 공장의 환경 센서(온도·습도·기압)와 AI 기반 이상 감지 결과(화재·낙하·구부러짐·이상소음)를 주기적으로 수집해야 한다. 시스템은 각 공장의 인프라 상태(노드 자원, 워크로드 상태, 장치 가용성)와 데이터 수집 지연 정보를 주기적으로 수집·모니터링해야 한다. 시스템은 센서 데이터, AI 결과, 인프라 상태, 파이프라인 상태를 종합하여 공장별 위험 수준을 정량적인 안전 점수로 산출해야 한다. 산출된 안전 점수는 100점 만점 기준으로 100에 가까울수록 안전, 0에 가까울수록 위험을 의미해야 한다 |
| **Alarm 전달 방법** | 시스템은 데이터 파이프라인의 지연·중단 상태를 단계별(정상/경고/위험)로 판정하고 Dashboard에 반영해야 한다. 공장 위험 상태가 변화하면 본사 관제 Dashboard에 준실시간으로 반영되어야 하며, 실시간 전달 경로 장애 시 주기적 조회 방식으로 보완되어야 한다. 시스템은 데이터 수신이 일정 시간 이상 중단된 장치 및 컴포넌트를 stale 상태로 판정하고 관제 담당자에게 명확히 표시해야 한다 |
| **모니터링 구축** | 시스템은 운영자용(인프라 메트릭 중심)과 본사 관제 담당자용(위험 현황 중심) 두 가지 모니터링 인터페이스를 분리해 제공해야 한다. 본사 관제 Dashboard는 공장별 위험 카드·환경 추세·인프라 현황·이상 시스템 목록·상태 변화 타임라인을 제공해야 한다. 인증된 사용자는 자신에게 허용된 공장 데이터만 조회할 수 있어야 하며, 미인가 공장 데이터 직접 호출은 차단되어야 한다. Dashboard는 현재 상태(실시간), 단기 이력, 장기 이력을 구분하여 조회할 수 있어야 한다. 시스템은 매일 정해진 시각에 공장 상태 요약과 주요 이벤트를 정리한 한국어 일간 보고서를 자동 생성하고, 권한 있는 사용자가 날짜·공장 기준으로 열람할 수 있어야 한다 |

## 4. 기타 요구사항

| 항목 | 내용 |
|------|------|
| **컨테이너 환경에 대한 가이드** | 운영형 Spoke의 워크로드는 특정 노드를 우선으로 스케줄링되어야 하며, 해당 노드 장애 시 자동으로 대체 노드에서 실행되어야 한다. 장애 복구 후 워크로드는 자동으로 우선 노드로 복귀(Failback)해야 한다. AI 추론 결과 스냅샷은 노드 장애 시에도 접근 가능해야 하며, 스토리지 동시 접근 충돌이 발생하지 않는 방식으로 보존되어야 한다. 엣지 로컬 데이터는 정해진 보존 기간이 지나면 자동으로 삭제되어 저장 공간을 관리해야 한다. 중앙 관제 환경의 단기 조회 데이터와 장기 보존 데이터, 일간 보고서 역시 정해진 보관 정책에 따라 자동으로 수명 주기가 관리되어야 한다. 테스트베드형 Spoke는 운영형과 동일한 데이터 발행 구조를 모의 데이터로 검증할 수 있어야 한다 |
| **기술내재화** | 인프라 구성, 클러스터 설정, 빌드 파이프라인, 배포 자동화, 데이터 수집·보존·조회 각 영역의 책임 경계를 명확히 분리하고, 각 영역의 역할과 운영 방법을 문서화하여 제공해야 한다 |
| **통합 검증 요건** | 시스템은 운영형 공장, 테스트베드형 공장, 장애(Failover), 롤백 시나리오에 더해 본사 관제 환경의 데이터 수집부터 위험 상태 산출, 사용자별 조회 범위 적용, 화면 갱신, 일간 보고서 열람까지의 종단 시나리오에 대한 통합 검증이 완료된 상태로 납품되어야 한다 |
