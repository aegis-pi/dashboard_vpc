### 2-2. 범위

| 범위 항목 | 내용 |
|-----------|------|
| **인프라 요구조건** | 클라우드 허브와 복수의 엣지 공장 Spoke를 연결하는 멀티 공장 인프라를 구축해야 한다. 제어/관리 영역(2번 Control / Management VPC)과 사용자 조회·데이터 처리 영역(1번 Data / Dashboard VPC)은 네트워크 및 권한 경계로 분리되어야 한다. 1번 VPC는 Public, Private App, Private Data 계층을 갖추고 ALB, ECS Fargate Backend, Redis, RDS PostgreSQL을 포함해야 한다 |
| **글로벌/관리형 서비스 요구조건** | 사용자 대시보드는 VPC 내부 자원만으로 구성하지 않고 CloudFront, WAF, Route53, ACM, Cognito, S3 정적 호스팅, IoT Core, DynamoDB, EventBridge, Bedrock, Secrets Manager, CloudWatch/X-Ray 등 VPC 밖 글로벌·관리형 서비스를 목적별로 사용해야 한다 |
| **Dashboard 요구조건** | 본사 관제 담당자용 Risk Twin Dashboard를 제공해야 한다. Dashboard는 Fleet Overview, Factory Detail, Environment, Infrastructure, Timeline, 일간 보고서 탭을 포함하고, 공장별 현재 위험도와 원인을 우선 표시해야 한다 |
| **사용자/권한 요구조건** | 사용자는 Cognito 기반 인증 후 RDS PostgreSQL의 사용자·공장·권한 관계에 따라 자신에게 허용된 공장 데이터만 조회할 수 있어야 한다. 전체 공장 조회, 단일 공장 조회, 보고서 조회, 사용자/알림룰 관리 범위는 역할별로 분리되어야 한다 |
| **데이터 요구조건** | Edge Agent가 IoT Core로 전송한 `factory_state`와 `infra_state`를 S3 raw에 원본 보존하고, Lambda data processor가 DynamoDB LATEST/HISTORY와 S3 processed를 갱신해야 한다. Dashboard는 현재 상태는 DynamoDB LATEST, 최근 그래프는 DynamoDB HISTORY, 상세/감사/리포트는 S3 processed와 reports prefix를 우선 조회해야 한다 |
| **실시간 요구조건** | DynamoDB Streams, Lambda notifier, Redis Pub/Sub, ECS WebSocket을 통해 공장 상태 변화가 Dashboard Web에 실시간으로 push되어야 한다. WebSocket 장애 시 30초 polling fallback을 제공해야 한다 |
| **보안 요구조건** | Dashboard는 Spoke K3s API, EKS admin API, ArgoCD admin API, Tailscale 관리망, S3 raw 전체 권한에 직접 접근할 수 없어야 한다. 모든 사용자 진입점은 HTTPS와 인증을 강제하고, Secret/Token/Private Key는 코드·문서에 기록하지 않아야 한다 |
| **CI/CD 요구조건** | 인프라·클러스터·빌드·배포 각 영역의 책임 경계를 명확히 분리하고, 수동 개입 없이 자동화된 파이프라인을 통해 변경 가능해야 한다. 클러스터 앱 상태 변경은 GitOps 경로를 사용해야 한다 |
| **운영/비용 요구조건** | 데모 운영 패턴에서는 `build-data-dashboard.sh`로 생성하고 `destroy-data-dashboard.sh`로 종료해 비용을 제어해야 한다. 새 AWS 상시 리소스 추가 시 비용 기준 문서가 함께 갱신되어야 한다 |

## 3. 사업 요구사항

### 3-1. 시스템 구축 요구사항

| 분류 | 항목 | 요구사항 내용 |
|------|------|--------------|
| **Infrastructure** | 전체 구조 | 시스템은 `factory-a/b/c` Spoke를 중앙 Hub와 Data / Dashboard VPC로 통합 관리할 수 있어야 한다. Control / Management VPC는 배포·제어·운영 관측을 담당하고, Data / Dashboard VPC는 데이터 처리 결과 조회와 사용자 Dashboard 제공을 담당해야 한다 |
| | 1번 VPC | 1번 Data / Dashboard VPC는 Public Subnet, Private App Subnet, Private Data Subnet으로 구성되어야 한다. Public Subnet에는 HTTPS ALB와 NAT Gateway 1개를 두고, Private App Subnet에는 ECS Fargate Dashboard Backend, ElastiCache Redis, Lambda notifier 접근 경계를 두며, Private Data Subnet에는 RDS PostgreSQL을 배치해야 한다 |
| | VPC 밖 글로벌/관리형 서비스 | Dashboard Web은 S3 정적 SPA와 CloudFront로 제공되어야 하며 S3 bucket은 public으로 열리지 않고 CloudFront OAC로만 접근되어야 한다. WAF는 외부 사용자 진입점 보호에 사용되어야 하고, Route53은 신규 Dashboard 도메인 위임과 `dashboard.<domain>`, `api.<domain>` alias를 관리해야 한다. ACM은 CloudFront용 us-east-1 인증서와 ALB용 regional 인증서를 분리해야 한다 |
| | 데이터 처리 경로 | Edge Agent는 AWS IoT Core로 상태 메시지를 전송해야 한다. IoT Rule은 원본을 S3 raw에 저장하고, Lambda data processor는 메시지를 정규화해 Risk Score, pipeline status, latest/history, processed 결과를 생성해야 한다. DynamoDB Streams는 저장 완료 이벤트를 발생시키고 Lambda notifier가 Redis Pub/Sub로 WebSocket fan-out을 트리거해야 한다 |
| | Server | 운영형 Spoke는 다중 노드 K3s로 구성되어 단일 노드 장애 시 서비스가 대체 노드에서 계속 실행되어야 한다. 테스트베드형 Spoke는 운영형과 동일한 데이터 발행 구조를 VM 환경에서 검증할 수 있어야 한다. Dashboard Backend는 FastAPI 기반 ECS Fargate 컨테이너로 운영되어 REST, WebSocket, RDS, DynamoDB, S3, Redis, Bedrock 연동을 제공해야 한다 |
| | Network | 허브와 각 공장 Spoke 간 제어 통신은 암호화된 전용 채널을 통해야 하며 사용자 Dashboard 접근망과 분리되어야 한다. Dashboard는 Tailscale/VPN에 직접 의존하지 않아야 하며, 사용자 접근은 CloudFront/ALB의 HTTPS 경로와 Cognito 인증을 통해서만 허용되어야 한다. VPC 간 직접 peering/TGW로 조회망과 제어망을 연결하지 않고 관리형 저장소와 IAM 권한을 통해 데이터 계약을 유지해야 한다 |
| | Container | 시스템은 선언적 GitOps 방식으로 공장별 컨테이너 워크로드를 중앙에서 배포·동기화·롤백할 수 있어야 한다. Dashboard Backend는 컨테이너 이미지로 빌드되어 ECR에 저장되고, ECS Service가 ALB target group 뒤에서 안정적으로 실행되어야 한다 |
| | Common | 소프트웨어 빌드·배포 파이프라인은 수동 개입 없이 자동화되어야 한다. 공장별 설정은 버전 관리 시스템에서 독립적으로 관리되어야 하며, 공장 추가 시 기존 구조를 변경하지 않고 확장 가능해야 한다 |

### 3-2. Dashboard 및 사용자별 조회 범위 요구사항

| 사용자/역할 | 조회 가능 범위 | 수행 가능 기능 | 제한 사항 |
|-------------|---------------|----------------|-----------|
| **본사 관제 관리자(Admin)** | 전체 공장 Fleet, 공장별 상세, Risk/환경/인프라 이력, Timeline, 일간 보고서, 사용자·공장·권한 메타데이터 | 전체 공장 위험 현황 확인, 사용자/권한 매핑 관리, 알림룰 설정, 감사 로그 확인, 임시 사용자 비활성화 | Dashboard에서 ArgoCD/EKS/Spoke API를 직접 제어하지 않음. 배포·클러스터 변경은 GitOps/운영 경로로만 수행 |
| **본사 관제 담당자(Operator)** | RDS `user_factory_access`에 매핑된 공장 목록과 해당 공장의 상세 화면, 보고서, 최근 이벤트 | 위험 공장 식별, 원인 확인, 보고서 열람, 운영 대응 판단 | 다른 공장 데이터와 사용자/권한 관리 화면은 조회 불가 |
| **현장 운영자(Local Operator)** | 자신이 담당하는 단일 공장 또는 지정 공장 그룹의 현재 상태, 장치/노드/워크로드 상태, Timeline | 현장 이상 여부 확인, 센서/장치/워크로드 이상 파악 | 전체 Fleet 통계, 타 공장 상세, 전역 사용자 관리, 클라우드 인프라 설정 조회 불가 |
| **배포 담당자/시스템 관리자** | 운영 관측 도구(Grafana, ArgoCD, CloudWatch 등)와 필요 시 지정된 Dashboard 공장 상태 | 배포 상태 확인, 장애 원인 분석, 운영 로그·메트릭 확인 | 사용자용 Dashboard가 운영 제어 plane이 되어서는 안 됨. Dashboard 권한은 별도 승인된 범위로 제한 |
| **발표/검토용 임시 사용자(Viewer)** | 지정된 공장 또는 데모용 전체 Fleet의 read-only 화면 | 화면 확인, 보고서 열람 | 쓰기, 권한 변경, 알림룰 변경, 운영 API 접근 불가. 시연 후 계정 disable 필요 |

| **Environment** | Risk Score, 온도, 습도, 기압, 화재/넘어짐/굽힘 AI score의 최근 1시간·2시간·24시간 추세를 표시해야 한다. 환경 그래프는 30초 단위 history bucket을 기본으로 사용해야 한다 | DynamoDB HISTORY#RISK, HISTORY#FACTORY |
| **Infrastructure** | 노드 Ready, CPU, Memory, Disk, workload 상태, restart count, 장치(BME280, Camera, Microphone), pipeline 상태를 표시해야 한다. 인프라 그래프는 20초 단위 history를 기준으로 해야 한다 | DynamoDB LATEST, HISTORY#INFRA |
| **Timeline** | Risk level 변화, 주요 원인 변화, pipeline status 변화, node ready 변화, workload 이상, device 이상을 시간순으로 표시해야 한다 | DynamoDB HISTORY, S3 processed |
| **일간 보고서** | 매일 09:00 KST에 생성된 공장별 Markdown 보고서를 조회하고 렌더링해야 한다. 보고서는 전일 24시간 Risk 추세, 주요 이벤트, pipeline 상태, 권장 조치를 포함해야 한다 | S3 reports, DynamoDB aegis-daily-report |

### 3-4. 데이터 및 저장소 요구사항

| 항목 | 요구사항 내용 |
|------|--------------|
| **S3 raw** | IoT Core 수신 원본 JSON은 `raw/{factory_id}/{source_type}/yyyy=...` 경로에 장기 보존되어야 한다. 원본 데이터는 감사, 재처리, 리포트 근거로 사용 가능해야 하며 Dashboard 기본 조회 경로가 되어서는 안 된다 |
| **S3 processed** | Lambda data processor의 Risk 계산 결과, 환경 상태 처리 결과, 인프라 상태 처리 결과는 `processed/` prefix에 장기 보존되어야 한다. Dashboard는 상세 이력, 감사, 리포트 입력에 processed 데이터를 사용할 수 있어야 한다 |
| **DynamoDB LATEST** | 공장별 현재 상태는 `pk=FACTORY#{factory_id}`, `sk=LATEST` 단일 item으로 유지되어야 한다. 최신 Risk, sensor, AI, infra, workload, device, pipeline status, dashboard summary를 빠르게 조회할 수 있어야 한다 |
| **DynamoDB HISTORY** | 최근 그래프용 short-term 시계열은 HISTORY item으로 저장되어야 한다. MVP 기본 TTL은 24시간이며, Risk/환경은 30초 bucket, 인프라는 20초 기준으로 조회 가능해야 한다 |
| **RDS PostgreSQL** | 사용자, 공장, 사용자-공장 권한, 알림룰, 감사 로그 같은 관계형 메타데이터는 RDS PostgreSQL에 저장해야 한다. Backend는 Cognito `sub`와 RDS app_user를 매핑하고 `user_factory_access`로 공장 목록을 필터링해야 한다 |
| **ElastiCache Redis** | Redis는 API 응답 캐시와 WebSocket Pub/Sub fan-out에 사용되어야 한다. `factory:LATEST:<factory_id>`, `factory:meta:<factory_id>`, `user:permissions:<user_id>` 같은 cache key에 TTL을 적용해야 한다 |
| **DynamoDB daily report** | 일간 보고서 메타데이터는 `aegis-daily-report` 테이블에 `report_date`, `factory_id`, `s3_key`, `summary`, `generated_at`, `model_id` 등을 저장해야 한다 |

### 3-5. 보안 요구사항

| 항목 | 요구사항 내용 |
|------|--------------|
| **인증** | Dashboard 사용자는 Cognito Hosted UI의 OIDC PKCE 흐름으로 로그인해야 한다. Self sign-up은 비활성화하고 관리자 생성 사용자만 허용해야 하며 MFA(TOTP)를 필수로 적용해야 한다 |
| **인가** | Backend는 Cognito JWT를 앱 레벨에서 검증하고, RDS PostgreSQL의 사용자·공장 권한 정보를 기준으로 API 응답을 필터링해야 한다. 권한 캐시는 Redis에 저장하되 TTL을 둬야 한다 |
| **시스템 감사** | 시스템은 공장에서 수집된 모든 원본 데이터를 변경 불가능한 원본 계층(S3 raw)에 보존하여 감사 추적이 가능해야 한다. 처리 결과와 보고서도 별도 prefix에 보존되어 재처리 결과와 비교 가능해야 한다 |
| **시스템 보안** | Dashboard Web/API는 Spoke K3s API, EKS admin API, ArgoCD admin API, Tailscale 관리망을 직접 조회할 수 없어야 한다. Dashboard API는 LATEST/HISTORY, processed, reports, 메타데이터에 필요한 최소 권한만 가져야 한다 |
| **네트워크 보안** | ALB와 CloudFront는 HTTPS만 허용해야 하며, RDS PostgreSQL은 ECS Fargate task 보안그룹에서만 5432 접근을 허용해야 한다. Redis는 Fargate task와 Lambda notifier 보안그룹에서만 접근해야 하며 transit encryption과 AUTH token을 사용해야 한다 |
| **Secret 관리** | DB 비밀번호, Redis AUTH token, Cognito client 정보, AWS credential, private key, token은 코드 저장소와 문서에 기록해서는 안 된다. Runtime Secret은 Secrets Manager와 환경변수 참조로 관리해야 한다 |
| **보안솔루션** | CloudFront/WAF는 Dashboard Web 진입점 보호에 사용되어야 하며, 간단한 SQL injection/XSS 패턴 차단 검증이 필요하다. 후속 Phase에서는 IdP federation, Security Hub, GuardDuty, Shield Advanced를 트리거 기반으로 검토해야 한다 |

### 3-6. 모니터링 및 실시간 요구사항

| 항목 | 요구사항 내용 |
|------|--------------|
| **Metric 정의** | 시스템은 각 공장의 환경 센서(온도·습도·기압), AI 기반 이상 감지 결과(화재·넘어짐·굽힘·이상소음), 노드 자원(CPU/Memory/Disk), workload 상태, device 상태, pipeline heartbeat를 수집해야 한다 |
| **Risk 산출** | 시스템은 센서 데이터와 AI 결과, stale 상태, pipeline 상태를 종합하여 0~100 Risk Score와 `안전 / 주의 / 위험` 상태를 산출해야 한다. 사용자는 숫자보다 상태와 주요 원인을 먼저 볼 수 있어야 한다 |
| **Alarm 전달 방법** | 데이터 파이프라인 상태는 `normal / warning / critical`로 판정되어 Dashboard에 반영되어야 한다. 일반 상태 변화는 통합 경로에서 10~35초 목표로 반영되어야 하며, DDB Streams 이후 WebSocket push는 1~2초 목표로 전달되어야 한다 |
| **Stale 판정** | 센서, Edge Agent, 노드, 카메라, 마이크, 데이터 파이프라인이 일정 시간 이상 무응답이면 stale 또는 확인 필요 상태로 표시해야 한다. stale 상태는 Risk 원인과 Timeline에 반영되어야 한다 |
| **모니터링 구축** | 운영자용 Grafana는 Hub/EKS/AMP/인프라 관측 도구로 유지하고, 사용자-facing 제품 화면은 Risk Twin Dashboard Web/API로 분리해야 한다. Dashboard는 Control / Management VPC API를 직접 조회하지 않아야 한다 |
| **관측성** | Backend, Lambda data processor, Lambda notifier, report-generator는 CloudWatch Logs와 구조화 로그를 남겨야 한다. X-Ray 또는 동등한 trace를 통해 ALB/ECS/DynamoDB/RDS/Redis/Bedrock 호출 흐름을 확인할 수 있어야 한다 |

### 3-7. LLM 보고서 요구사항

| 항목 | 요구사항 내용 |
|------|--------------|
| **일간 보고서 생성** | EventBridge Scheduler는 매일 09:00 KST에 report-generator Lambda를 실행해야 한다. Lambda는 지난 24시간의 DDB HISTORY와 S3 processed 데이터를 집계하고 Bedrock Claude 3 Haiku를 호출해 한국어 Markdown 보고서를 생성해야 한다 |
| **보고서 저장** | 생성된 보고서는 `reports/<YYYY-MM-DD>/<factory_id>.md` 경로에 저장되어야 하며, 메타데이터는 DynamoDB `aegis-daily-report`에 기록되어야 한다 |
| **보고서 조회** | Dashboard는 보고서 탭에서 날짜와 공장 기준으로 보고서를 조회하고 Markdown으로 렌더링해야 한다. 권한이 없는 공장의 보고서는 조회할 수 없어야 한다 |
| **보안 제약** | Bedrock 프롬프트에는 Secret, token, private key, MFA OTP, 계정 세부정보를 포함해서는 안 된다. Bedrock 호출 IAM은 필요한 모델 InvokeModel 권한과 보고서 생성에 필요한 최소 데이터 읽기 권한으로 제한해야 한다 |

## 4. 기타 요구사항

| 항목 | 내용 |
|------|------|
| **컨테이너 환경에 대한 가이드** | 운영형 Spoke의 워크로드는 특정 노드를 우선으로 스케줄링되어야 하며, 해당 노드 장애 시 자동으로 대체 노드에서 실행되어야 한다. 장애 복구 후 워크로드는 조건에 따라 우선 노드로 복귀(Failback)해야 한다. Dashboard Backend 컨테이너는 non-root 이미지, healthcheck, CloudWatch Logs, circuit breaker, 최소 IAM Task Role을 갖추어야 한다 |
| **기술내재화** | Terraform은 AWS 인프라, Ansible은 bootstrap·설정·검증, GitHub Actions는 CI·이미지 빌드·push, GitHub+ArgoCD는 CD의 source of truth로 문서화되어야 한다. Data / Dashboard VPC 운영 절차와 build/destroy 절차도 운영 문서로 제공해야 한다 |
| **통합 검증 요건** | 시스템은 Edge Agent → IoT Core → S3 raw → Lambda data processor → DynamoDB LATEST/HISTORY → DDB Streams → Lambda notifier → Redis → WebSocket → Dashboard Web의 end-to-end 경로를 검증해야 한다. 인증 실패 401, 권한 없는 공장 조회 차단, WebSocket push, 일간 보고서 생성, WAF 차단, Backend p95 < 500ms, WebSocket 100 concurrent connection을 검증해야 한다 |
| **비용 관리 요건** | Phase 1 상시 운영과 데모 운영 비용을 구분해 산정해야 한다. NAT Gateway, ALB, ECS Fargate, RDS PostgreSQL, Redis, Bedrock, CloudFront, S3, DynamoDB, Cognito, Route53 비용 항목을 비용 기준 문서와 일치시켜야 한다 |
| **후속 확장 요건** | 메시지율, DDB HISTORY 비용, 시계열 쿼리 지연, 동시 사용자 수, 컴플라이언스 요구가 임계값을 넘으면 Phase 2~4에서 Timestream, Kinesis, OpenSearch, Multi-AZ, SageMaker/EKS GPU, IdP federation, Security Hub, GuardDuty, PrivateLink, Athena/Glue를 ADR 기반으로 검토해야 한다 |
